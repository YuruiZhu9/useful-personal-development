    AI Coding 的原则

    模型已经足够聪明，你只需要做好引导

    01

    需求拆解

    具体到你需要的是什么

    02

    约束明确

    明确模型不能做的事

    03

    验收标准

    设定规则和目标

    04

    控制长度

    不要一次给太多输入

    先把意图写清楚：目标、边界、技术栈、输入输出、约束、验收标准。先把 Spec 写得清晰明确，比纯 vibe coding 更有效。

    三 · AI Coding 经验2 / 4

    Vibe Coding 方法论

    不是"凭感觉让AI乱写"，而是用自然语言驱动、用中间产物约束、用验收标准收敛

    需求分析
    →
    架构设计
    →
    任务规划
    →
    代码生成
    →
    资源生成
    →
    代码集成

    核心要点

    • 意图明确——"我要什么"说清楚
    • 约束越清楚，成功率越高
    • 技术栈、目录结构、接口协议越明确越好
    • Vibe 只是交互模式变了，工程化才是落地关键

    过渡到 Spec Coding

    Vibe Coding → TDD → SDD

    使用结构化文档规定：

    • 项目目录结构、构建命令
    • 测试流程、代码规范
    • PR 提交流程

    三 · AI Coding 经验4 / 4

    使用小技巧

  • 多开终端：任务拆开并发处理（可延伸到 git feature 分支并行开发）
  • 遇到 bug 直接丢给 CC：环境变量、Skills 加载失败、MCP 启动失败——把报错和上下文丢给 CC
  • "生成 → 检查 → 修正 → 再验收"：不要追求一次生成可行的代码
  • 长时间 auto：使用 --dangerously-skip-permissions
  • Spec & Plan：Spec 是"写什么、怎么写、设计是啥"；Plan 是"按什么顺序写、写到什么阶段测试"
  • 思路不明确时：先用 Brainstorming skill 做头脑风暴，或与 LLM 交互形成大致思路
  • 迭代式开发：写 Spec → CC 生成 → 调整 Spec → CC 再生成 → 反复直到满意

    Agent = Model + Harness

    Model 只是"大脑"，Harness 才是把模型变成"能稳定做事的系统"的工程骨架

    Task Context

    当前任务的 prompt 与输入信息

    Project Context

    代码仓库结构与工程规则

    Environment Context

    工具、权限与执行环境

    CC 的 Harness 体系：Hooks · Skills · Subagent · MCP · Memory · Plan/Task

    环境上下文决定了 Agent 能够做什么，以及能够做到什么程度。

    四 · Harness 与协同2 / 4

    CC vs OpenClaw

    Claude Code

    专用编码 Agent

    • 定位：专门的 coding agent
    • 写代码、改代码、调代码、跑任务
    • Harness 体系完善

    OpenClaw

    智能助手 / 调度层

    • 定位：智能中枢
    • 接收消息、定时触发、唤醒调度
    • 自主唤醒能力，原生适配通讯工具

    OpenClaw 属于单独的一类 agent，帮助打造智能中枢。需要花时间让 agent 具备足够的 skill（使用有冷启动，类似于基建类游戏）。

    四 · Harness 与协同3 / 4

    OpenClaw Agent Workspace 核心文件

    文件作用
    AGENTS.mdAgent 的工作说明书与行为准则，划定核心职责、任务流程与行为边界
    BOOTSTRAP.md全新工作区的一次性初始化引导脚本，完成后需删除
    SOUL.mdAgent 的灵魂内核与顶层宪法，定义核心身份、人格、价值观与安全红线
    MEMORY.mdAgent 的长期记忆中枢，存储用户稳定偏好、重要决策等跨会话信息
    HEARTBEAT.mdAgent 的生物钟与主动任务引擎，负责周期任务、健康巡检与主动提醒
    user.md记录用户个人信息、职业背景、沟通偏好等完整档案
    identity.md定义 Agent 基础身份、运行环境配置、工具权限等

    六 · 总结

    工程实践思维

    01 自然语言的模糊性和执行的精确性，是阻碍长程工作的一大障碍

    Vibe Coding → Spec Coding

    02 工具的差异不只是模型差异，更是 Harness 和 Workflow 的差异

    03 真正决定效果的，是如何使用 Spec、Skills，并约束和验收你的项目

    04 CC 适合作为主线工具，但最好和其他工具协同使用
